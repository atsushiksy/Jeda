1. Introduction

Jeda is a concurrent programming language designed for modeling and verifying electronics chip/system designs.  The language provides and not limited to:
  • flexible and dynamic concurrent programming mechanism with time/cycle concept
  • object oriented programming
  • various system classes such as semaphore and events to construct dynamic concurrent systems efficiently.
  • aspect oriented programming
The language syntax is based on C and Verilog also borrowing some ideas from C++, Java and Perl.  A Jeda program consists of various functions, and it starts the execution from the function "main".  Here's an example of a simple Jeda code. (I'm not kidding..)

  void main() {
    printf( "Hello world!\n" ) ;
  }

The basic data elements that the language supports are:
    int            -- integer
    bit            -- Verilog style multi-value (can hold x & z state) bit.
                            It can be vectored as bit [5:0].
    string       -- String
    float         -- Single precision floating variable
    double     -- Double precision floating variable
    signal       -- Store the pointer to the port (interface to Verilog)

   The user defined data type can be declared as 'class'.

Various array forms are supported over the data element above. The following example provides declaration of all the array forms supported in Jeda for type int.

    int  foo[5] ;           // Array
    int  moo[5][4][2]  // Multidimensional array
    int  bar[] ;             // Associative array
    int  boo[*] ;          // string indexed array
    int  bee<> ;           // list

Jeda provides safeguards to minimize user programming errors. For instance, the Jeda compiler  employs strict type checks.  In languages line C and C++, pointers, cast assign are usually the areas programming errors occur, thus Jeda does not support them. It  checks the data consistency at runtime (unless optimized) to detect the programming errors that cannot be detected  during compile time.

Jeda implements an automatic garbage collection system.  Due to availability of this functionality, the user can now create and forget the variable elements without worrying about the memory leak(as similarly found in Java).  The lack of the similar garbage collection mechanism makes many languages unsuitable for describing complex logic behavior as they  require lots of programming effort  to keep track  of memory usage(to avoid memory leaks that could hamper long simulation runs). This added burden also  affects the one's productivity as time is wasted  instead of writing system models or verification code.

Jeda is a "compiled" language in the sense that a user's code is compiled into a binary form at the execution time. The Jeda compiler (jedacmp) compiles the Jeda code into intermediate object file. Jeda liner (jedalnk) links multiple object files and create a C code. This C code is compiled and linked with Jeda runtime library to a dynamic library form (.dl) as the compiled execution image.

Jeda runs with Verilog as a PLI code and accesses the Verilog signals via a module (Jeda module). This module contains the ports the user specified in Jeda code and drive/sample can be  done through these ports. The user create an instance of this Jeda module in the Verilog environment and run Verilog with the Jeda PLI code.

Jeda PLI code consists of two parts, static part and dynamic part. The static part is a common code that is statically linked with Verilog (or compiled with VCS). The dynamic part is produced as the result of Jeda code compile & link process as mentioned above, and dynamically loaded to Verilog at execution time. So, as far as the port signals are identical, the user can use the compiled Verilog execution image as is, and run different test cases by just specifying the dynamic library with the plus argument ( +jeda_code=<lib_name> ) .
