3. テスト プラン

設計検証を、テストプランを作成してから始めることは、良い習慣だと思います。 テストプランは、どのようなチェックメカニズムがテストベンチに必要かを示して くれ、ドライバーや基礎的なモジュール(クラス等)を設計する際に、多くの ガイダンスを与えてくれます。

この例では、実際の設計検証で行なうような完全なテストケースを示すわけでは なく、最小限のもののみがあげられています。(Verilogモデルが、単純な 動作モデルなので、多くのコーナーケースをチェックするようなリストを 作成するのは、どのみち困難です。)

  • マネージメント インターフェース テスト

    • マネージメント インターフェース をテストし、レジスタの内容にアクセスを行なう。
  • セル入力スライド テスト

    • 複数のセルを異なったポートから、タイミングをスライドさせて送り、 色々な入力タイミングでの競合ケースをテストする。
  • セル出力ポート渋滞テスト

    • 出力ポートが混雑した状態を作り、出力ポートの競合によるフロー制御機能を テストする。
  • 低速トランスミッター テスト

    • tx側の転送速度が遅く、セルが渋滞するケースをテストする。 tx側からのフロー制御機能を確認する。
  • エラーセル テスト

    • エラーセルを挿入し、正しく破棄されることを確認する。
  • 飢餓状態 テスト

    • ポート間のアービトレーションがフェアでないために発生する飢餓状態を つくり出す。ポート０以外は、転送が行なわれないはず。
  • ランダム テスト

    • パラメータを変更可能なランダムなテストシーケンスを提供する。
このテストリストより、以下のようなメカニズムがテストベンチに必要であることが 導き出されます。
  • 複数のポートに対して並列的に、セルを色々なタイミングでドライブ出来ること。
  • 出力ポートの競合状態をつくり出すメカニズム
  • 出力側の転送速度を制御するメカニズム
  • エラーセルを挿入するメカニズム
  • ランダムテストをサポートするメカニズム
最初のメカニズムを実現するのは、Jedaの並列プログラミング機能を用いて 容易に行なうことが出来ます。その他のメカニズムは、テストベンチの 色々な場所で実現されています。以下の章で、説明します。
