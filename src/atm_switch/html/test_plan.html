3. Test Plan

It is always good to start your verification work with the test plan. It shows the fundamental mechanisms for the self-checking testbench and it will provide a lot of guidance in developing the drivers and other primitive modules.

I'm not intend to provide a full set of test cases I usually do in my verification work. Here, I listed a minimal set of test cases. (As the verilog model is a simple behavioral one,  I can't come up with many corner cases anyway.)

  • management if test

    • testing management interface and access to the register contents
  • cell input slide test

    • send multiple cells from different ports in sliding timing window, testing conflict case in the various input timing
  • cell output port congestion test

    • create the output port congestion situation, testing the output conflict case to check the flow control mechanism
  • slow transmitter speed test

    • test the case that tx side is slow and cell is congested, testing the flow control mechanism from tx side.
  • error cell test

    • inject error cell to check it's correctly dropped
  • starvation test

    • create the design problem case that the arbitration per port is not fair, and other than port0 won't be serviced
  • random test

    • provides the configurable random test sequence
This list shows that we need the following mechanisms:
  • ability to drive a cell to a port in various timing, concurrently to multiple port
  • mechanism to create output port conflict situation
  • mechanism to control the transmit side speed
  • mechanism to inject error cells
  • mechanism for random
First mechanism is the easy one, as Jeda provide the concurrent programming primitives. Other mechanisms are implemented in various places in the testbench. The following chapters will show them.
